# Seed Decoding — Подтверждённая спецификация

## Метаданные

| Поле | Значение |
|------|----------|
| Функция | `FUN_009eb6b0` |
| Статус | ✅ Подтверждено |
| Источник | Ghidra decompiler, `isaac-ng.exe` |
| Дата исследования | 06.07.2026 |
| Архитектура | x86 (32-bit) |

---

## Краткое описание

`FUN_009eb6b0` — функция **декодирования пользовательского seed** в внутренний `uint32`.

> ⚠️ **Это НЕ RNG.** Функция не генерирует случайные числа. Она не использует LCG. Она не поддерживает внутреннее состояние генератора.

---

## Входные данные

### Формат строки

Пользователь вводит строку длиной **9 символов**:

```
XXXX XXXX
```

Где:
- 4 символа
- пробел (разделитель)
- 4 символа

Функция проверяет:
- Общую длину строки = 9
- Наличие пробела на позиции 4 (индекс 4)
- Корректность всех остальных символов

---

## Алфавит

```
ABCDEFGHJKLMNPQRSTWXYZ01234V6789
```

Это Base32-подобный алфавит. Из него **исключены неоднозначные символы**:

| Исключён | Причина |
|----------|---------|
| `I` | Слишком похож на `L` |
| `O` | Слишком похож на `0` |
| `U` | Слишком похож на `V` |
| `5` | Слишком похож на `S` |

> ⚠️ Символы `W`, `1`, `2`, `6`, `8` **входят** в алфавит (ранее ошибочно указаны как исключённые).

Итого: **32 допустимых символа** → каждый кодирует ровно **5 бит**.

## Таблица декодирования

Функция строит **lookup table** размером 256 байт.

| Вход | Выход |
|------|-------|
| Допустимый ASCII-символ | Значение 0..31 |
| Недопустимый символ | `0xFF` |

Lookup table обеспечивает **O(1)** декодирование каждого символа.

---

## Этапы декодирования

### Этап 1: Декодирование символов

Каждый из 8 символов (позиции 0,1,2,3,5,6,7,8; позиция 4 — пробел) преобразуется
в число 0..31 (5 бит) через lookup table. Недопустимый символ → возврат 0.

### Этап 2: Bit packing

8 декодированных 5-битных значений (v0..v7) упаковываются в 32-битный seed:

```
packed = ((((((v0 << 5 | v1) << 5 | v2) << 5 | v3) << 5 | v4) << 5 | v5) << 2 | (v6 >> 3))
```

Распределение 40 бит (8 символов × 5 бит):

| Биты | Источник | Назначение |
|------|----------|------------|
| 31..27 | v0 (5 бит) | Seed |
| 26..22 | v1 (5 бит) | Seed |
| 21..17 | v2 (5 бит) | Seed |
| 16..12 | v3 (5 бит) | Seed |
| 11..7 | v4 (5 бит) | Seed |
| 6..2 | v5 (5 бит) | Seed |
| 1..0 | v6 >> 3 (2 бита) | Seed |
| 7..5 | v6 & 0x07 (3 бита) | Checksum |
| 4..0 | v7 (5 бит) | Checksum |

> **Важно:** 7-й символ (v6) разделён — старшие 2 бита идут в seed, младшие 3 бита в checksum. 8-й символ (v7) целиком входит в checksum. Все 8 символов значимы.

### Этап 3: XOR

После упаковки выполняется XOR с константой:

```
seed = packed ^ 0x0FEF7FFD
```

> Это **первая подтверждённая константа** преобразования seed.

### Этап 4: Checksum

Контрольная сумма (8 бит) вычисляется из XOR'нутого значения seed:

```
checksum = 0
temp = seed
while temp != 0:
    byte_val = temp & 0xFF
    checksum = ROL8((checksum + byte_val) & 0xFF, 1)
    temp >>= 5
```

Где `ROL8(x, 1)` — циклический сдвиг 8-битного значения влево на 1 бит.

Ожидаемый checksum: `((v6 << 5) | v7) & 0xFF`.

При несовпадении функция возвращает 0.

Следовательно, пользовательский seed содержит **встроенную проверку корректности**.

## Возвращаемое значение

Функция возвращает `uint32`.

Это **внутренний seed**, который затем используется системой RNG.

```
Строка seed (XXXX XXXX)
    ↓
Декодирование (FUN_009eb6b0)
    ↓
Internal uint32 seed
    ↓
RNG (FUN_009EB880 — инициализация)
```

## Полная диаграмма

```
Пользователь вводит:
  XXXX XXXX
  (8 значащих символов + пробел-разделитель)
    ↓
Проверка формата (длина 9, пробел на позиции 4)
    ↓
Построение lookup table (256 байт)
    ↓
Base32 decode (каждый символ → 5 бит, всего 40 бит)
    ↓
Bit packing (v0..v5 + v6>>3 → 32 бита seed)
    ↓
XOR 0x0FEF7FFD
    ↓
Checksum verification (v6&0x07 + v7 → 8 бит)
    ↓
Internal uint32 seed
    ↓
[следующая стадия — FUN_009EB880]
    ↓
RNG initialization
```

## Что это открытие меняет

Теперь **доказано** существование отдельного слоя декодирования пользовательского seed перед системой генерации случайных чисел.

Это означает:

1. **Пользовательский seed ≠ внутренний seed.** Это разные представления одного и того же значения.
2. **Поиск RNG необходимо продолжать уже после преобразования строки в `uint32`.** Функция `FUN_009eb6b0` — это только декодер, а не генератор.
3. **Функция `FUN_009eb6b0` больше не должна классифицироваться как RNG.** Ранее в документации могло утверждаться, что найден RNG — это исправлено. Найден именно декодер строкового представления seed.
4. **Существование checksum** означает, что игра валидирует корректность seed до его использования, что упрощает отладку и тестирование.

## Подтверждённые этапы обработки seed

На данный момент подтверждены следующие стадии:

| № | Стадия | Статус |
|---|--------|--------|
| 1 | `Пользовательский seed (строка)` → `FUN_009eb6b0` → `Internal uint32 seed` | ✅ Подтверждено |
| 2 | `Internal uint32 seed` → `FUN_009eb880` → `RNG initialization` (shift: 3/17/25, fallback FUN_006eeF60 при seed==0) | ✅ Подтверждено |
| 3 | `RNG (инициализирован)` → `FUN_006ef890` → `Next PRNG value` (XORSHIFT Marsaglia) | ✅ Подтверждено |
| 4 | Источник первоначального `StartSeed` при создании нового забега | 🔴 Не найден |
