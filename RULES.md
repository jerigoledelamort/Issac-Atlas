# Правила разработки Isaac Atlas

## Основные принципы

### 1. Исследование прежде кода

> «Сначала понимание, потом реализация»

- Запрещено писать код на основе неподтверждённых гипотез
- Каждое изменение должно ссылаться на исследование в `research/`
- Статус гипотезы должен быть «Подтверждено» перед использованием в коде

### 2. Документация precedes реализация

> «Незадокументированное знание — потерянное знание»

- Исследование документируется до начала реализации
- API специфицируется до написания кода
- Изменения в архитектуре фиксируются в ADR

### 3. Точность воспроизведения

> «Мы не создаём новую игру — мы воспроизводим существующую»

- Запрещено «улучшать» алгоритмы оригинальной игры
- Требуется 100% совпадение с оригиналом
- Любые отклонения документируются как известные различия

### 4. Разделение ответственности

> «Engine не знает о Atlas. Atlas не знает о внутренностях Engine»

- Engine не зависит от Atlas
- Atlas использует Engine через публичный API

## Обязательные требования

### Для коммитов

- [ ] Сообщение коммита следует формату (см. ниже)
- [ ] Код проходит линтер
- [ ] Код проходит форматирование
- [ ] Добавлены тесты для новой функциональности
- [ ] Обновлена документация при изменении API

### для Pull Request

- [ ] PR ссылается на исследование (если применимо)
- [ ] PR ссылается на ADR (если изменяет архитектуру)
- [ ] Все тесты проходят
- [ ] Код ревьюирован минимум одним разработчиком
- [ ] Обновлён CHANGELOG.md (для изменений пользователя)

### Для исследований

- [ ] Гипотеза чётко сформулирована
- [ ] Методология описана
- [ ] Данные собраны и задокументированы
- [ ] Выводы обоснованы
- [ ] Статус установлен

## Формат сообщений коммитов

```
<тип>(<область>): <описание>

[опциональное тело]

[опциональный футер]
```

### Типы коммитов

| Тип | Описание |
|-----|----------|
| `feat` | Новая функциональность |
| `fix` | Исправление бага |
| `docs` | Документация |
| `research` | Новое исследование |
| `refactor` | Рефакторинг без изменения поведения |
| `test` | Тесты |
| `chore` | Вспомогательные изменения |
| `ci` | Изменения CI/CD |
| `build` | Изменения системы сборки |
| `perf` | Улучшение производительности |
| `style` | Форматирование, стиль кода |
| `arch` | Архитектурные изменения |

### Области

- `engine` — изменения в Isaac Engine
- `atlas` — изменения в Isaac Atlas
- `research` — исследования
- `docs` — документация
- `tests` — тесты
- `ci` — CI/CD
- `build` — сборка

### Примеры

```
feat(engine): добавить модуль верификации RNG
fix(atlas/cli): исправить парсинг аргументов --seed
docs(research): обновить документацию по генерации этажей
research(rng): добавить анализ seed'а версии 1.06
arch(engine): переименовать database/ в storage/
```

## Ветвление

### Стратегия

- `main` — основная ветка, всегда стабильна
- `develop` — ветка разработки (опционально)
- `feature/<name>` — новые возможности
- `research/<name>` — исследования
- `fix/<name>` — исправления
- `docs/<name>` — документация
- `release/<version>` — подготовка релиза

### Именование веток

```
feature/engine-rng-module
research/floor-generation-algorithm
fix/atlas-cli-seed-parsing
docs/architecture-update
release/0.1.0
```

## Код ревью

### Обязательные аспекты

1. **Корректность** — код делает то, что должен
2. **Стиль** — следует STYLE_GUIDE.md
3. **Тесты** — покрывают новую функциональность
4. **Документация** — API задокументировано
5. **Производительность** — нет очевидных проблем

### Запрещено в ревью

- Личные комментарии
- Требования к стилю, не описанные в STYLE_GUIDE.md
- Изменение области PR без согласия автора

## Статусы гипотез

| Статус | Описание | Можно ли использовать в коде |
|--------|----------|------------------------------|
| `Новая` | Только предложена | ❌ Нет |
| `В проверке` | Требуется верификация | ❌ Нет |
| `Подтверждено` | Проверена и принята | ✅ Да |
| `Отвергнуто` | Не подтвердилась | ❌ Нет |

## Нарушения правил

### Критические

- Использование неподтверждённых гипотез в коде
- Изменение алгоритмов оригинальной игры без документирования
- Коммит секретов или чувствительных данных

**Последствие:** Немедленное исправление, обсуждение в команде.

### Серьёзные

- Отсутствие тестов для новой функциональности
- Изменение API без обновления документации
- Игнорирование комментариев ревью

**Последствие:** Запрос на исправление, блокировка мержа.

### Незначительные

- Опечатки в документации
- Нарушение стиля кода
- Неидеальные сообщения коммитов

**Последствие:** Комментарий в ревью, исправление по возможности.
