# Вклад в проект Isaac Atlas

## Принципы вклада

Проект Isaac Atlas строится на строгих принципах научного подхода к reverse engineering.

---

## Обязательные правила

### Запрещено ❌

1. **Использовать неподтверждённые гипотезы в коде**
   - Гипотезы должны быть проверены перед использованием
   - Статус гипотезы должен быть "Подтверждено"

2. **Писать генератор до завершения исследования**
   - Сначала документация в `research/`
   - Потом реализация в `engine/`

3. **Изменять алгоритмы оригинальной игры**
   - Наша цель — точное воспроизведение
   - Не "улучшаем", не "оптимизируем", воспроизводим

### Обязательно ✅

1. **Каждое исследование документируется**
   - Все находки фиксируются в `research/`
   - Документация precedes код

2. **Каждая гипотеза получает статус**
   - `Новая` — только что предложена
   - `В проверке` — требует верификации
   - `Подтверждено` — проверена и принята
   - `Отвергнуто` — не подтвердилась

3. **Сначала доказательство, потом код**
   - Код без документации не принимается
   - Реализация следует спецификации

---

## Процесс вклада

### 1. Исследование

```
research/
└── <тема>/
    ├── hypothesis.md       # Описание гипотезы
    ├── evidence.md         # Доказательства
    ├── tests.md            # Тестовые данные
    └── status.md           # Статус гипотезы
```

### 2. Документирование

Создайте документацию в соответствующей директории:
- `research/` — для исследований
- `docs/` — для общей документации

### 3. Реализация

Только после завершения исследования и документирования:
- Код следует документации
- Код комментируется ссылками на исследования

### 4. Тестирование

- Тесты должны проверять совпадение с оригиналом
- Тестовые данные документируются

### 5. Pull Request

PR должен содержать:
- [ ] Ссылку на исследование
- [ ] Ссылку на документацию
- [ ] Описание изменений
- [ ] Результаты тестов

## Структура исследований

### Шаблон исследования

```markdown
# Название исследования

## Статус
- Статус: Новая / В проверке / Подтверждено / Отвергнуто
- Дата начала: YYYY-MM-DD
- Автор: @username

## Гипотеза
Описание предполагаемого поведения

## Методология
Как проводилось исследование

## Данные
Собранные эмпирические данные

## Анализ
Интерпретация данных

## Выводы
Подтверждается ли гипотеза

## Ссылки
- Связанные исследования
- Внешние источники
```

## Стиль кода

### C++ (Engine)

- Следовать современному C++ (C++17/20)
- Избегать неопределённого поведения
- Документировать все публичные API
- Ссылаться на исследования в комментариях

### Общий стиль

- Чётные имена переменных и функций
- Комментарии объясняют "почему", а не "что"
- Модули следуют архитектуре

## Коммиты

### Формат сообщений

```
<тип>(<область>): <описание>

[опциональное тело]

[опциональный футер]
```

### Типы коммитов

- `feat` — новая функциональность
- `fix` — исправление бага
- `docs` — документация
- `research` — новое исследование
- `refactor` — рефакторинг
- `test` — тесты
- `chore` — вспомогательные изменения

### Примеры

```
research(rng): добавить анализ seed'а
docs(architecture): обновить диаграмму зависимостей
feat(engine): добавить модуль верификации
```

## Ветвление

- `main` — основная ветка
- `feature/<name>` — новые возможности
- `research/<name>` — исследования
- `fix/<name>` — исправления
- `docs/<name>` — документация
