## Краткое описание

`FUN_009eb6b0` — функция **декодирования пользовательского строкового seed'а** во внутренний `uint32`.

> ⚠️ **Это НЕ RNG.** Функция не генерирует случайные числа. Она только преобразует строковое представление сида в числовое.

---

## Формат строкового сида

Пользователь вводит строку длиной **9 символов**:

```
XXXX XXXX
```

Где:
- Позиции 0-3: 4 символа алфавита
- Позиция 4: пробел (обязательный разделитель)
- Позиции 5-8: 4 символа алфавита

### Проверки формата

Функция проверяет:
1. Общую длину строки = 9
2. Наличие пробела на позиции 4 (индекс с нуля)
3. Корректность всех остальных символов (входят в алфавит)

При любой ошибке функция возвращает `0`.

## Алфавит

```
ABCDEFGHJKLMNPQRSTWXYZ01234V6789
```

### Характеристики

- **32 символа** → каждый кодирует ровно **5 бит** (значения 0..31)
- Base32-подобный алфавит с исключением неоднозначных символов

### Исключённые символы

| Символ | Причина исключения |
|--------|-------------------|
| `I` | Слишком похож на `L` |
| `O` | Слишком похож на `0` |
| `U` | Слишком похож на `V` |
| `5` | Слишком похож на `S` |

> Примечание: символ `W` **входит** в алфавит (позиция 18).

## Алгоритм декодирования

### Этап 1: Построение lookup table

Функция строит таблицу поиска размером 256 байт:

```c
memset(local_110, 0xff, 0x100);
for (uVar4 = 0; uVar4 < 0x20; uVar4++) {
    local_110["ABCDEFGHJKLMNPQRSTWXYZ01234V6789"[uVar4]] = (byte)uVar4;
}
```

- Все байты инициализируются `0xFF` (признак недопустимого символа)
- Для каждого символа алфавита записывается его индекс (0..31)
- Lookup обеспечивает **O(1)** декодирование каждого символа

### Этап 2: Декодирование 8 символов

Строка обрабатывается посимвольно (позиции 0..8):

| Позиция | Символ | Выходной индекс | Переменная |
|---------|--------|-----------------|------------|
| 0 | char[0] | 0 | `local_10[0]` |
| 1 | char[1] | 1 | `local_10[1]` |
| 2 | char[2] | 2 | `local_10[2]` |
| 3 | char[3] | 3 | `local_10[3]` |
| 4 | ` ` (пробел) | пропускается | — |
| 5 | char[5] | 4 | `local_c` |
| 6 | char[6] | 5 | `local_b` |
| 7 | char[7] | 6 | `local_a` |
| 8 | char[8] | 7 | `local_9` |

Для каждой позиции:
1. Символ lookup'ится в таблице
2. Если результат `0xFF` → возврат `0` (невалидный символ)
3. Иначе значение сохраняется в выходной массив

### Этап 3: Упаковка битов

8 декодированных 5-битных значений упаковываются в 32-битное число:

```
packed = ((((((v0 << 5 | v1) << 5 | v2) << 5 | v3) << 5 | v4) << 5 | v5) << 2 | (v6 >> 3))
```

Где:
- `v0..v5` — полные 5-битные значения (30 бит)
- `v6 >> 3` — старшие 2 бита 7-го значения (2 бита)
- **Итого: 32 бита**

### Этап 4: XOR с константой

После упаковки применяется XOR:

```c
seed = packed ^ 0x0FEF7FFD;
```

**Константа:** `0x0FEF7FFD` = 267 354 109 (десятичное)

Это **первая подтверждённая константа** преобразования seed.

### Этап 5: Вычисление checksum

Контрольная сумма вычисляется из XOR'нутого значения seed:

```c
checksum = 0;
for (temp = seed; temp != 0; temp >>= 5) {
    byte_val = temp & 0xFF;           // младший байт (как signed char)
    checksum = ROL8((checksum + byte_val) & 0xFF, 1);
}
```

Где `ROL8(x, 1)` — циклический сдвиг 8-битного значения влево на 1 бит:

```c
ROL8(x, 1) = ((x << 1) | (x >> 7)) & 0xFF
```

В декомпилированном коде это выражено как:

```c
cVar2 = bVar3 + (char)uVar6;
bVar3 = cVar2 * '\x02' - (cVar2 >> 7);  // эквивалентно ROL8(cVar2, 1)
```

**Особенности:**
- Цикл обрабатывает seed порциями по 5 бит (сдвиг `>> 5`)
- На каждой итерации берётся младший **байт** (8 бит), а не 5 бит
- Для 32-битного seed выполняется 7 итераций (32 / 5 ≈ 6.4 → 7)
- При `seed == 0` цикл не выполняется, `checksum = 0`

### Этап 6: Проверка checksum

Ожидаемое значение checksum собирается из двух декодированных значений:

```c
expected = ((v6 << 5) | v7) & 0xFF;
```

Где:
- `v6` (7-й символ) — младшие 3 бита идут в checksum (биты 5-7)
- `v7` (8-й символ) — все 5 бит идут в checksum (биты 0-4)

**Распределение битов 7-го символа:**
- Старшие 2 бита (`v6 >> 3`) → идут в seed (этап 3)
- Младшие 3 бита (`v6 & 0x07`) → идут в checksum (этап 6)

**Итоговое распределение 40 бит (8 символов × 5 бит):**
| Биты | Назначение |
|------|------------|
| 31..0 | Seed (32 бита) |
| 7..5 | Checksum bits 7..5 (из v6) |
| 4..0 | Checksum bits 4..0 (из v7) |

### Этап 7: Возврат результата

```c
if (checksum == expected) {
    return seed;  // валидный сид
}
return 0;  // невалидный сид
```

## Возвращаемое значение

Функция возвращает `uint32` — внутренний seed, который затем используется системой RNG.

### Неоднозначность возврата 0

**Важное ограничение алгоритма:**

Возврат `0` **неоднозначен**:
- `0` может означать **валидный seed = 0**
- `0` может означать **ошибку** (невалидная строка)

Это связано с тем, что функция не имеет отдельного флага ошибки и использует `0` как для валидного сида 0, так и для обозначения ошибки.

Для однозначной проверки валидности в реализации `seed_decoder.py` предусмотрена функция `decode_seed_safe()`, возвращающая структуру с явным флагом `is_valid`.

## Алгоритм кодирования (обратное преобразование)

Кодирование — обратная операция к декодированию. Каждому `uint32` соответствует ровно одна валидная строка.

### Этапы кодирования

1. **Отменить XOR:** `pre_xor = seed ^ 0x0FEF7FFD`

2. **Извлечь 5-битные значения из `pre_xor`:**
   ```python
   v0 = (pre_xor >> 27) & 0x1F
   v1 = (pre_xor >> 22) & 0x1F
   v2 = (pre_xor >> 17) & 0x1F
   v3 = (pre_xor >> 12) & 0x1F
   v4 = (pre_xor >> 7)  & 0x1F
   v5 = (pre_xor >> 2)  & 0x1F
   v6_top2 = pre_xor & 0x3  # 2 старших бита v6
   ```

3. **Вычислить checksum из seed:**
   ```python
   checksum = _compute_checksum(seed)
   ```

4. **Извлечь младшие биты из checksum:**
   ```python
   v6_bottom3 = (checksum >> 5) & 0x07  # биты 5-7
   v7 = checksum & 0x1F                  # биты 0-4
   ```

5. **Собрать v6:**
   ```python
   v6 = (v6_top2 << 3) | v6_bottom3
   ```

6. **Отобразить в символы алфавита:**
   ```python
   chars = [ALPHABET[v] for v in [v0, v1, v2, v3, v4, v5, v6, v7]]
   return f"{chars[0]}{chars[1]}{chars[2]}{chars[3]} {chars[4]}{chars[5]}{chars[6]}{chars[7]}"
   ```

**Декодирование:**

| Позиция | Символ | Значение |

| 0 | B | 1 |
| 1 | M | 11 |
| 2 | R | 15 |
| 3 | 9 | 31 |
| 4 | (пробел) | — |
| 5 | K | 9 |
| 6 | S | 16 |
| 7 | H | 7 |
| 8 | Y | 20 |

**Упаковка:**
```
packed = ((((((1 << 5 | 11) << 5 | 15) << 5 | 31) << 5 | 9) << 5 | 16) << 2 | (7 >> 3))
       = 182449344 = 0x0ADFF4C0
```

**XOR:**
```
seed = 0x0ADFF4C0 ^ 0x0FEF7FFD = 0x05308B3D = 87067453
```

**Checksum:**
```
temp = 0x05308B3D
iter 1: byte=0x3D=61, checksum=ROL8(61,1)=122, temp>>=5
iter 2: byte=0x59=89, checksum=ROL8(211,1)=167, temp>>=5
iter 3: byte=0x22=34, checksum=ROL8(201,1)=147, temp>>=5
iter 4: byte=0x61=97, checksum=ROL8(244,1)=233, temp>>=5
iter 5: byte=0x53=83, checksum=ROL8(60,1)=120, temp>>=5
iter 6: byte=0x02=2, checksum=ROL8(122,1)=244, temp>>=5=0
checksum = 244 = 0xF4
```

**Ожидаемый checksum:**
```
expected = ((7 << 5) | 20) & 0xFF = (224 | 20) & 0xFF = 244
```

## Ограничения и особенности

### 1. Неоднозначность seed = 0

Как описано выше, возврат `0` не позволяет отличить валидный seed от ошибки.

**Решение:** Использовать `decode_seed_safe()` для явной проверки валидности.

### 2. Все uint32 кодируемы

Любое значение `uint32` (0..4294967295) может быть закодировано в валидную строку.

### 3. Не все строки валидны

Строка проходит валидацию только если:
- Длина ровно 9 символов
- Пробел на позиции 4
- Все символы входят в алфавит
- Checksum совпадает

### 4. Регистрозависимость

Алфавит содержит только заглавные буквы. Строчные буквы не входят в алфавит и приводят к ошибке декодирования.
